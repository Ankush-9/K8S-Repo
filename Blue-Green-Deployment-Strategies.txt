# Blue-Green-Deployment Strategies :
~ It is one of the strategies used the industry for deployment for the purpose of releasing applications
~ We do have other strategies for deployment
~ There are different opinions on this strategy as it’s an old methodology but still used in some firms

~ Advantages:
-> It reduces risk and minimizes application’s down time

*When our application is live to the world we term it as deployment

~ Every app has multiple releases where with every release new features, bug fixes ,security updates and so are appended to the existing app
~ For a heavy application, it takes some time to deploy the newer versions of the app 
~ The time may vary from minutes to hours depending on the size of the app

~ The re-developed or new features requires creating of new pod
~ The older pods needs to be deleted while new are required to be created and added
~ For the process of deletion and re-deployment of pods our app will be down —> App becomes un-accessible
~ Also, suppose if latest Pods have some issues? —> new release has some major bugs
~ We cannot keep both the old and new bods simultaneously

# Solution to this!
~ To Solve the above issues we use the Blue-Green-Deployment Strategies
~ We use two production Environments known as blue and green environments

~ In both the environments we have shared resources(like database), App(pods) and Services
~ We have router to route traffic 
~ As the end user tries to access the application and while the application is live in an environment —> An environment (out of the two either blue or green) is active to the world(end users) while the other is present only for testing purpose
~ The thing is after the testing is over so now, the new app needs not to be deployed just a small change in service exposure
~ Within few seconds the end users will be routed to the new application
~ Also the old env is not deleted at all ,it’s present in the other environment
~ If any issue occurs in the new release the traffic can be re-routed to the old environment while the team fixes the issue

~ Further if the new release is functional normally, the old environment is deleted and further as per requirement a new environment is created with new releases and bug fixes and same steps are followed

~ In the steps illustrated above the blue-green deployment strategy is used in the industry

*We can call the active env as green while the old as blue and vice versa 
*We do have other strategies also, but few advantages are related to this strategy which are :
	—> Zero Downtime 
	—> Rapid Releases
	—> Easy rollbacks
	—> Good customer experience 

#Alternatives to Blue-Green-Deployment

—>Canary Deployment(best for risk mitigation)
	Canary Deployment releases the new version to a small subset of users first(the “canary” group).Traffic is gradually increased to the new version as confidence grows, allowing for real world testing and early detection of bugs.
—>Shadow Deployment
	Shadow deployment (or "dark launching") is a deployment strategy where a new version of an application or AI model runs alongside the current production version, receiving a copy of live traffic but not returning results to users. 	It allows for real-world testing, performance validation, and bug detection without impacting actual user experience.
 —> A/B Testing
	It is similar to Canary Deployment but traffic is routed based on user characteristics(e.g. location, cookies , device) to test different features/UI/workflows.
 —> Recreate
	Shuts down all the old pods before creation of new pods	

*We have so many deployment strategies available with K8S which are used as per demand
*Blue-Green is an old deployment which was used for a longer stretch of time

# Blue-Green-Deployment Practical:
*The first release is the blue deployment
*live-service is to deploy our application
*pre-prod yml file is to access the application of green deployment because we are just switching the pods
	—>Entire traffic is not routed to green deployment i.e. blue is live to the world
	—>Initially the green deployment is used by the testing team only

-> blue-deployment.yml
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app-deployment
spec:
  replicas: 2
  strategy:
    type: RollingUpdate
  selector:
    matchLabels:
      app: java-web-app
      version: v1
      color: blue
  template:
    metadata:
      labels:
        app: java-web-app
        version: v1
        color: blue
    spec:
      containers:
      - name: javawebappcontainer
        image: hacker123shiva/springbt-in-docker:latest
        imagePullPolicy: Always
        ports:
        - containerPort: 8080
...

->green-deployment.yml
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app-deployment-green
spec:
  replicas: 2
  strategy:
    type: RollingUpdate
  selector:
    matchLabels:
      app: java-web-app
      version: v2
      color: green
  template:
    metadata:
      labels:
        app: java-web-app
        version: v2
        color: green
    spec:
      containers:
      - name: javawebappcontainer
        image: hacker123shiva/springbt-in-docker:latest
        imagePullPolicy: Always
        ports:
        - containerPort: 8080
...

->live-service.yml
---
apiVersion: v1
kind: Service
metadata:
  name: web-app-live-service
spec:
  type: LoadBalancer
  selector:
    app: java-web-app
    color: green
  ports:
  - port: 80
    targetPort: 8080
...

->pre-prod.yml
---
apiVersion: v1
kind: Service
metadata:
  name: web-app-prod-service
spec:
  type: NodePort
  selector:
    app: java-web-app
    color: green
  ports:
    - port: 80
      targetPort: 8080
      protocol: TCP
      nodePort: 31725
...

—> kubectl apply -f blue-deployment.yml
—> kubectl apply -f live-service.yml
—> kubectl apply -f green-deployment.yml
—> kubectl apply -f pre-prod.yml

*The only changes to be made to switch from blue—>green or vice versa is to use  the live-service yml file to make changes
—>The only change to be made is to change the color only 

*Make changes in the non-live deployment apply them using “kubectl apply -f blue-deployment.yml”
*As changes are made in the live-deployment we have to reapply them using “kubectl apply -f live-service.yml”
